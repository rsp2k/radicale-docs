## Overview

Beyond the core CalDAV and CardDAV specifications, Radicale implements numerous extension RFCs that add modern features like calendar colors, managed attachments, availability tracking, and enhanced task management. This page documents our excellent compliance with these extended specifications.

<Aside type="tip">
Radicale's extended RFC support often exceeds that of commercial CalDAV servers, providing cutting-edge features to open-source users.
</Aside>

## Modern iCalendar Extensions

### RFC 7986 - New Properties for iCalendar

**Status**: ✅ **Fully Supported (100% compliance)**

RFC 7986 adds modern properties to iCalendar, including calendar colors, images, and video conferencing details.

<CardGrid>
  <Card title="COLOR Property" icon="seti:config">
    Assign colors to calendars and events for visual organization. Supported by all modern clients.
  </Card>

  <Card title="IMAGE Property" icon="document">
    Attach images to calendar events (logos, locations, photos). Display-friendly URLs.
  </Card>

  <Card title="CONFERENCE Property" icon="rocket">
    Video conferencing links (Zoom, Teams, Meet) with proper formatting and labels.
  </Card>
</CardGrid>

**Key Features:**
- ✅ Full property preservation through scheduling
- ✅ Client compatibility with Apple Calendar, Thunderbird, etc.
- ✅ No data loss during sync operations

**Example:**

```icalendar
BEGIN:VEVENT
UID:meeting@example.com
DTSTART:20250120T140000Z
DTEND:20250120T150000Z
SUMMARY:Team Meeting
COLOR:blue
IMAGE:https://example.com/logo.png
CONFERENCE;VALUE=URI;FEATURE=VIDEO:https://meet.example.com/abc-123
END:VEVENT
```

### RFC 8607 - CalDAV Managed Attachments

**Status**: ✅ **Complete (95% compliance)**

RFC 8607 defines server-managed attachments for calendar events, allowing files to be stored on the CalDAV server with proper access control.

<CardGrid>
  <Card title="Attachment Storage" icon="document">
    Store event attachments (PDFs, images, documents) on the CalDAV server.
  </Card>

  <Card title="Managed URLs" icon="rocket">
    ATTACH property with managed URLs instead of embedded data.
  </Card>

  <Card title="Access Control" icon="approve-check">
    Attachments inherit event permissions for secure sharing.
  </Card>
</CardGrid>

**Key Features:**
- ✅ Server-side attachment storage
- ✅ Access control for attachments
- ✅ Efficient storage (no base64 bloat)
- ✅ Multiple attachments per event

**Example:**

```bash
# Upload attachment
curl -X PUT \
  -u alice:password \
  -H "Content-Type: application/pdf" \
  --data-binary @presentation.pdf \
  https://server/alice/calendar.ics/meeting.ics?attachment=presentation.pdf

# Event with managed attachment
BEGIN:VEVENT
UID:meeting@example.com
SUMMARY:Project Review
ATTACH:https://server/alice/calendar.ics/meeting.ics?attachment=presentation.pdf
END:VEVENT
```

### RFC 7953 - Calendar Availability (VAVAILABILITY)

**Status**: ✅ **Complete (95% compliance)**

RFC 7953 adds VAVAILABILITY component for defining when users are available for scheduling, separate from actual calendar events.

<CardGrid>
  <Card title="VAVAILABILITY Component" icon="seti:clock">
    Define available periods separate from actual events.
  </Card>

  <Card title="AVAILABLE Periods" icon="approve-check">
    Specify when you're available for meetings.
  </Card>

  <Card title="Free/Busy Integration" icon="magnifier">
    Free/busy queries respect availability rules.
  </Card>
</CardGrid>

**Key Features:**
- ✅ VAVAILABILITY component support
- ✅ AVAILABLE and UNAVAILABLE periods
- ✅ Free/busy query integration
- ✅ Recurrence rule support for availability

**Example:**

```icalendar
BEGIN:VCALENDAR
BEGIN:VAVAILABILITY
UID:availability@example.com
DTSTART:20250101T000000Z
DTEND:20251231T235959Z
BEGIN:AVAILABLE
UID:work-hours@example.com
DTSTART:20250101T090000Z
DTEND:20250101T170000Z
RRULE:FREQ=WEEKLY;BYDAY=MO,TU,WE,TH,FR
SUMMARY:Work Hours
END:AVAILABLE
END:VAVAILABILITY
END:VCALENDAR
```

## Enhanced Task Management

### RFC 9253 - Enhanced VTODO

**Status**: ✅ **Complete (95% compliance)**

RFC 9253 extends VTODO with task hierarchy and dependency relationships, enabling sophisticated task management.

<CardGrid>
  <Card title="Task Hierarchy" icon="list-format">
    RELATED-TO property for parent-child task relationships.
  </Card>

  <Card title="Task Dependencies" icon="star">
    Define task dependencies (predecessor/successor).
  </Card>

  <Card title="Relationship-Based Queries" icon="magnifier">
    Query tasks by relationship type and hierarchy.
  </Card>
</CardGrid>

**Key Features:**
- ✅ RELATED-TO with RELTYPE parameter
- ✅ PARENT relationship type
- ✅ DEPENDS-ON relationship type
- ✅ Task filtering by relationship
- ✅ Hierarchical task structures

**Example:**

```icalendar
BEGIN:VTODO
UID:project@example.com
SUMMARY:Complete Project
RELATED-TO;RELTYPE=CHILD:task-1@example.com
RELATED-TO;RELTYPE=CHILD:task-2@example.com
STATUS:IN-PROCESS
END:VTODO

BEGIN:VTODO
UID:task-1@example.com
SUMMARY:Design Phase
RELATED-TO;RELTYPE=PARENT:project@example.com
RELATED-TO;RELTYPE=DEPENDS-ON:task-2@example.com
STATUS:COMPLETED
END:VTODO

BEGIN:VTODO
UID:task-2@example.com
SUMMARY:Requirements Gathering
RELATED-TO;RELTYPE=PARENT:project@example.com
STATUS:COMPLETED
END:VTODO
```

## WebDAV Base Protocol

### RFC 4918 - WebDAV

**Status**: ⚠️ **Partial (85% compliance)**

RFC 4918 defines the base WebDAV protocol. Radicale implements most features with some intentional gaps.

**Implemented Methods:**
- ✅ **GET** - Retrieve resources
- ✅ **PUT** - Create/update resources
- ✅ **DELETE** - Remove resources
- ✅ **PROPFIND** - Query properties
- ✅ **PROPPATCH** - Update properties
- ✅ **MKCOL** - Create collections
- ✅ **MOVE** - Move resources
- ✅ **OPTIONS** - Discover capabilities

**Not Implemented:**
- ❌ **LOCK/UNLOCK** - Locking not implemented (most CalDAV clients don't require it)
- ⚠️ **COPY** - Partial implementation (works for some cases)

<Aside type="note">
The lack of LOCK/UNLOCK is intentional. Modern CalDAV clients use ETags and conditional operations instead, which Radicale fully supports.
</Aside>

## iTIP Protocol

### RFC 5546 - iTIP (Transport-Independent Interoperability Protocol)

**Status**: ✅ **Core Complete (95% compliance)**

RFC 5546 defines iTIP, the protocol for calendar scheduling messages. This is the foundation for RFC 6638 (CalDAV scheduling) and RFC 6047 (iMIP).

**Implemented Methods:**
- ✅ REQUEST - Meeting invitations
- ✅ REPLY - Attendee responses
- ✅ CANCEL - Cancellations
- ✅ COUNTER - Counter proposals
- ✅ DECLINECOUNTER - Decline counters
- ✅ REFRESH - Request updates

**Validation Features:**
- ✅ SEQUENCE number validation
- ✅ RECURRENCE-ID support
- ✅ DELEGATED-TO/DELEGATED-FROM
- ✅ PARTSTAT tracking
- ✅ METHOD-specific property requirements

## Quota Management

### RFC 4331 - Quota and Size Properties

**Status**: ✅ **Complete (90% compliance)**

RFC 4331 defines WebDAV properties for quota management, allowing clients to check storage limits.

**Key Features:**
- ✅ `quota-available-bytes` property
- ✅ `quota-used-bytes` property
- ✅ Per-user quota enforcement
- ✅ PROPFIND quota queries

**Example:**

```bash
curl -X PROPFIND \
  -u alice:password \
  -H "Content-Type: application/xml" \
  -d '<?xml version="1.0"?>
      <D:propfind xmlns:D="DAV:">
        <D:prop>
          <D:quota-available-bytes/>
          <D:quota-used-bytes/>
        </D:prop>
      </D:propfind>' \
  https://server/alice/
```

## Synchronization

### RFC 6578 - WebDAV Sync

**Status**: ✅ **Complete (90% compliance)**

RFC 6578 defines efficient collection synchronization using sync tokens, avoiding full re-sync operations.

**Key Features:**
- ✅ sync-collection REPORT
- ✅ Sync tokens for incremental sync
- ✅ Efficient change detection
- ✅ Deleted item tracking

**Example:**

```bash
# Initial sync
curl -X REPORT \
  -u alice:password \
  -H "Content-Type: application/xml" \
  -d '<?xml version="1.0"?>
      <D:sync-collection xmlns:D="DAV:">
        <D:sync-token/>
        <D:sync-level>1</D:sync-level>
        <D:prop><D:getetag/></D:prop>
      </D:sync-collection>' \
  https://server/alice/calendar.ics/

# Subsequent syncs use returned token
curl -X REPORT \
  -u alice:password \
  -H "Content-Type: application/xml" \
  -d '<?xml version="1.0"?>
      <D:sync-collection xmlns:D="DAV:">
        <D:sync-token>https://server/sync/abc123</D:sync-token>
        <D:sync-level>1</D:sync-level>
        <D:prop><D:getetag/></D:prop>
      </D:sync-collection>' \
  https://server/alice/calendar.ics/
```
